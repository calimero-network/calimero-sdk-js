Why QuickJS?

Pros

Full TypeScript support · npm ecosystem · Proven by NEAR · Smaller than V8/SpiderMonkey

Cons

~450 KB overhead vs ~50 KB AssemblyScript · ~2-5× slower than native WASM

We chose QuickJS over AssemblyScript for the significantly better developer experience — full TypeScript generics, npm packages, and familiar tooling.

Build Pipeline

Method Call Flow

What happens when a JSON-RPC call reaches the node:

1

JSON-RPC arrives

Calimero node receives the call with JSON parameters for the target method.

2

WASM loads

Node loads the service WASM module, creates QuickJS runtime, loads bytecode + ABI.

3

Parameters converted

JSON → ABI-compatible types: string bigints → BigInt, number arrays → Uint8Array, etc.

4

Method executes

Your TypeScript logic runs. env.* host function calls go to the Calimero runtime.

5

CRDT delta tracked

Collection mutations (map.set, vec.push) accumulate in the Rust delta context.

6

State persisted

ABI-aware Borsh serialization: [version: u8=1][state: borsh][collections: legacy].

7

Delta flushed

flush_delta() turns recorded actions into a causal delta. commit() finalises the execution.

8

Network broadcast

Delta broadcast to all peers. Receiving nodes apply the causal delta and run event handlers.

Method Parameters (Host → QuickJS)

Host sends JSON. Dispatcher converts using the embedded ABI:

→ Uint8Array (bytes)
object
→ Map (if ABI says map type)
unknown JSON object

Return Values (QuickJS → Host)

ABI-compatible types converted before JSON serialisation:

BigInt
→ string (safe integer transport)
Uint8Array
→ number[] (JSON-compatible bytes)
Map

CRDT Collection Encoding

Collections are stored as lightweight handles — not full copies:

// Serialized handle in storage (not the actual data)
{ "__calimeroCollection": "Vector", "id": "caf3…hex…" }

// Real CRDT state is keyed by the 32-byte ID in Rust storage
// Mutations issue incremental host calls — no deep clone on access

Rust SDK vs JS SDK

Rust services run storage collections inside the host runtime. QuickJS services run in an isolated VM and must round-trip through host calls:

env.persist_root_state()

Stores the serialised root document through the same storage interface Rust uses, so Merkle hashes and CRDT actions stay aligned.

env.flush_delta()

Turns recorded host-side actions into a causal delta — equivalent to what Rust does automatically after method return.

env.commit()

Reports execution result. Both Rust and JS SDKs must call this exactly once per execution for receipts, events, and broadcasts.

These extra steps exist because QuickJS cannot mutate host data structures directly. The JS SDK produces the same Merkle roots, artifacts, and deltas that the core runtime expects from the Rust SDK.

Security & Sandboxing

WASM Sandbox

QuickJS runs inside the WASM sandbox. No filesystem access, no network access — only approved host functions.

Host Validation

All host functions validate inputs: buffer bounds, register validation, type checking.

Rust Compatibility

JS services produce the same serialisation format as Rust services — they can run on the same network and sync state.

Events

calimero-sdk-js / events

Overview

Events enable async communication between nodes. When a node emits an event, it is broadcast via the P2P network. Receiving nodes apply the delta and execute the named handler — the author node does not execute its own handlers.

Author node

Emits event → broadcasts delta to network. Does NOT run its own handler.

Receiving nodes

Receive event → apply delta → execute handler → update state.

Handlers may run concurrently

Two events can arrive at the same node in any order. Handlers MUST be commutative, idempotent, independent, and pure.

Defining Events

Use the @Event decorator. Fields are serialised with ABI-aware Borsh for Rust compatibility:

import { Event } from '@calimero-network/calimero-sdk-js';

@Event
export class ItemAdded {
  constructor(
    public key: string,
    public value: string,
    public timestamp: number
  ) {}
}

@Event
export class ItemRemoved {
  constructor(public key: string) {}
}

Emitting Events

Without handler

Broadcast event — no handler executed on receiving nodes:

import { emit } from '@calimero-network/calimero-sdk-js';

emit(new ItemAdded('k', 'v', Date.now()));

With handler

Receiving nodes execute the named method after applying the delta:

import { emitWithHandler } from '@calimero-network/calimero-sdk-js';

emitWithHandler(
  new ItemAdded('k', 'v', Date.now()),
  'onItemAdded'
);

Event Handlers

Handler methods live in the @Logic class. They are called by name when the event arrives at a peer:

@Logic(MyApp)
export class MyAppLogic extends MyApp {
  addItem(key: string, value: string): void {
    this.items.set(key, value);
    emitWithHandler(new ItemAdded(key, value, Date.now()), 'onItemAdded');
  }

  // Runs on RECEIVING nodes only
  onItemAdded(event: ItemAdded): void {
    this.itemCount.increment(); // safe: G-Counter
  }
}

Handler Requirements

Handlers may execute in parallel and in any order. They must satisfy all four properties:

1. Commutative — order-independent

✅ Safe

onUserRegistered(e: UserRegistered): void {
  this.userCount.increment(); // G-Counter
}

❌ Unsafe

onCreate(e: Created): void {
  this.items.set(e.id, 'created');
}
onUpdate(e: Updated): void {
  // Assumes onCreate ran first!
  const cur = this.items.get(e.id); // ❌
}

2. Independent — no shared state between handlers

✅ Safe

handlerA(): void {
  this.counters.set('a', createCounter());
}
handlerB(): void {
  this.counters.set('b', createCounter()); // different key
}

❌ Unsafe

handlerA(): void {
  this.shared.set('count', 1); // RACE!
}
handlerB(): void {
  this.shared.set('count', 2); // RACE!
}

3. Idempotent — safe to retry

✅ Safe

handler(e: ItemAdded): void {
  this.items.set(e.id, 'processed'); // same result each time
}

❌ Unsafe

handler(amount: number): void {
  externalAPI.charge(amount); // may charge twice!
}

4. Pure — no side effects

✅ Safe

handler(data: string): void {
  this.items.set(data, 'processed');
  env.log('Processed'); // logging OK
}

❌ Unsafe

handler(email: string): void {
  httpClient.post('/notify', email); // external!
}

Best Practices

Keep handlers simple

// ✅ GOOD — single, focused update
onUserRegistered(): void {
  this.registrationCount.increment();
}

// ❌ BAD — too complex, unsafe side effects
onUserRegistered(user: User): void {
  this.createProfile(user);
  this.sendEmail(user); // ❌
  this.notifyAdmins(); // ❌
}

Use multiple targeted events

// ✅ Decompose into atomic events
registerUser(user: User): void {
  this.users.set(user.id, user);

  emit(new UserRegistered(user.id));
  emit(new ProfileNeeded(user.id));
  emit(new WelcomeEmailNeeded(user.email));
}

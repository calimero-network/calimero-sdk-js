Migration Guide

calimero-sdk-js / migration — Rust → JavaScript

Side-by-Side Comparison

Rust SDK

use calimero_sdk::app;
use calimero_storage::collections::UnorderedMap;

// State
#[app::state]
#[derive(BorshSerialize, BorshDeserialize)]
pub struct KvStore {
  items: UnorderedMap<String, String>,
}

// Logic
#[app::logic]
impl KvStore {
  #[app::init]
  pub fn init() -> KvStore {
    KvStore { items: UnorderedMap::new() }
  }

  pub fn set(&mut self, key: String, val: String) {
    self.items.insert(key, val).unwrap();
  }

  pub fn get(&self, key: &str) -> Option<String> {
    self.items.get(key).unwrap()
  }
}

JavaScript SDK

import { State, Logic, Init } from '@calimero-network/calimero-sdk-js';
import { UnorderedMap } from '@calimero-network/calimero-sdk-js/collections';

// State
@State
export class KvStore {
  items: UnorderedMap<string, string> = createUnorderedMap();
}

// Logic
@Logic(KvStore)
export class KvStoreLogic extends KvStore {
  @Init
  static initialize(): KvStore {
    return new KvStore();
  }

  set(key: string, val: string): void {
    this.items.set(key, val);
  }

  get(key: string): string | null {
    return this.items.get(key);
  }
}

Key Differences

1. Decorators vs Macros

Rust

#[app::state]

JavaScript

@State

Rust

#[app::logic]

JavaScript

@Logic(StateClass)

Rust

#[app::init]

JavaScript

@Init

Rust

#[app::event]

JavaScript

@Event

2. Error Handling

Rust — explicit Result<T, E>

pub fn set(&mut self, key: String) -> app::Result<()> {
  self.items.insert(key, val)?;
  Ok(())
}

JavaScript — collections throw on error

set(key: string): void {
  this.items.set(key, val); // throws on error
}

3. Initialization

Rust — struct literal

KvStore {
  items: UnorderedMap::new(),
}

JavaScript — inline field initializer

@State
export class KvStore {
  items = createUnorderedMap(); // ← inline!
}

Always initialise inline so the runtime rehydrates persisted CRDT IDs. Constructor-based init runs on every invocation and would create fresh (empty) collections.

Collection Mapping

→ UnorderedMap<K, V> — same API
Vector<T>
→ Vector<T> — same API

→ Counter — same behavior
LwwRegister<T>
→ LwwRegister<T> — same API
UnorderedSet<T>
→ UnorderedSet<T> — same API

Cross-SDK Compatibility

JavaScript and Rust services can:
✅ Run on same network
✅ Sync state via CRDTs
✅ Emit / receive events
✅ Call each other (xcall)

Both SDKs use ABI-aware Borsh serialization — same wire format.

Common Pitfalls

Forgetting inline field initialization

❌ BAD

@State
export class MyApp {
  // Not initialised! Breaks on hydration.
  items: UnorderedMap<string, string>;
}

✅ GOOD

@State
export class MyApp {
  items = createUnorderedMap<string, string>();
}
Putting methods in the @State class

❌ BAD

@State
export class MyApp {
  items = createUnorderedMap();
  set(key: string) { // ❌ in @State
    this.items.set(key, 'v');
  }
}

✅ GOOD

@State
export class MyApp {
  items = createUnorderedMap();
}

@Logic(MyApp)
export class MyAppLogic extends MyApp {
  set(key: string) { // ✅ in @Logic
    this.items.set(key, 'v');
  }
}

Migration Checklist

✅ Replace macros with decorators
#[app::state] → @State, etc.
✅ Update initialization
Struct literals → inline field factories
✅ Remove ? operator
JS collections throw directly
✅ Update imports
@calimero-network/calimero-sdk-js
✅ Test service
Verify all endpoints work correctly
✅ Verify multi-node sync
CRDT deltas must match Rust SDK format

When to Use Each SDK

Use Rust SDK when…

  • Performance is critical
  • Building complex algorithms
  • Need smallest WASM size (~100 KB)
  • Team has Rust expertise

Use JavaScript SDK when…

  • Rapid prototyping
  • Team knows TypeScript
  • npm ecosystem access needed
  • Developer experience is priority

Both SDKs are fully compatible and can interact on the same network.

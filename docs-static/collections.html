CRDT Collections

calimero-sdk-js / collections

Overview

Calimero provides conflict-free replicated data types (CRDTs) for automatic state sync across all peers. Values are serialised with ABI-aware Borsh, so data written from JavaScript matches bytes produced by Rust services as long as both sides share the same schema.

Collection Types

UnorderedMap<K, V>

Last-Write-Wins map for key-value storage. Conflict resolution: higher timestamp wins.

import { UnorderedMap } from '@calimero-network/calimero-sdk-js/collections';

const map = new UnorderedMap<string, string>();

map.set('key', 'value');
const val = map.get('key'); // 'value'
const has = map.has('key'); // true
map.remove('key');

map.entries(); // [['k', 'v'], …]
map.keys(); // ['k1', 'k2', …]
map.values(); // ['v1', 'v2', …]
// Nested collections work automatically!
const nested = new UnorderedMap<string, UnorderedSet<string>>();

const tags = new UnorderedSet<string>();
tags.add('urgent');
nested.set('task:123', tags);

// Mutations propagate automatically
const existing = nested.get('task:123');
existing?.add('high-priority'); // ✅ just works

Automatic Nested Collection Tracking

The SDK automatically detects and propagates changes in nested collections. No manual re-serialization needed.

Before (manual)

// ❌ Had to force re-serialization
const reactionMap = map.get(msgId);
if (reactionMap) {
  const users = reactionMap.get(emoji)
    || new UnorderedSet();
  users.add(userId);
  reactionMap.set(emoji, users); // ⬅️ manual
  map.set(msgId, reactionMap); // ⬅️ manual
}

Now (automatic)

// ✅ Just write natural code
function addReaction(msgId, emoji, userId) {
  let reactions = map.get(msgId)
    || new UnorderedMap();
  map.set(msgId, reactions);

  let users = reactions.get(emoji)
    || new UnorderedSet();
  reactions.set(emoji, users);

  users.add(userId); // propagates!
}

Vector<T>

Ordered list that maintains insertion order.

import { Vector } from '@calimero-network/calimero-sdk-js/collections';

const vec = new Vector<string>();

vec.push('first');
vec.push('second');
const item = vec.get(0); // 'first'
const len = vec.len(); // 2
const last = vec.pop(); // 'second'

UnorderedSet<T>

Last-Write-Wins set for unique membership.

import { UnorderedSet } from '@calimero-network/calimero-sdk-js/collections';

const set = new UnorderedSet<string>();

set.add('alice'); // true on first insert
set.add('alice'); // false when present
const has = set.has('alice'); // true
set.delete('alice');
const all = set.toArray(); // []
const sz = set.size(); // 0

Counter

Grow-only G-Counter. Each node tracks its own count; total is the sum across all nodes.

import { Counter } from '@calimero-network/calimero-sdk-js/collections';

const ctr = new Counter();

ctr.increment();
ctr.increment();
const total = ctr.value(); // 2n

Distributed Counting

Node A increments: local_A = 1
Node B increments: local_B = 1
Total = local_A + local_B = 2
No conflict — each node owns its slice.

LwwRegister<T>

Last-Write-Wins register for single values. Higher timestamp wins on conflict.

import { LwwRegister } from '@calimero-network/calimero-sdk-js/collections';

const reg = new LwwRegister<string>();

reg.set('hello');
const val = reg.get(); // 'hello'
const ts = reg.timestamp(); // nanoseconds

UserStorage<T>

User-owned, signed storage. Keys are 32-byte public keys. Writes are signed by the executor and verified on other nodes — prevents impersonation.

import { createUserStorage } from '@calimero-network/calimero-sdk-js';

interface Profile { name: string; score: number; }

const profiles = createUserStorage<Profile>();

// Insert for current executor (auto key)
profiles.insert({ name: 'Alice', score: 100 });

// Get own data
const mine = profiles.get();

// Get another user by public key
const pub = new Uint8Array(32);
const other = profiles.getForUser(pub);

profiles.remove(); // remove own data
profiles.entries(); // [[pubKey, profile], …]
profiles.size(); // number of users

How It Works

Writing creates a StorageType::User action signed by the executor's identity key. Receiving nodes verify the signature and a strictly-increasing nonce to prevent replay attacks.

Use Cases

  • Per-user settings and preferences
  • User-owned game data (scores, inventory)
  • Personal documents with ownership proof
  • Any verifiably user-owned data

FrozenStorage<T>

Immutable, content-addressable storage. Values are keyed by their SHA256 hash. Once inserted, values cannot be updated or deleted.

import { FrozenStorage, FrozenValue } from '@calimero-network/calimero-sdk-js/collections';
import { createFrozenStorage } from '@calimero-network/calimero-sdk-js';

const docs = createFrozenStorage<Document>();

// Add returns SHA256 hash of content
const hash = docs.add({ title: 'Spec', body: '…' });

const doc = docs.get(hash); // retrieve by hash
const exists = docs.has(hash); // true
const all = docs.entries(); // [[hash, doc], …]

// Compute hash without storing
const h = FrozenStorage.computeHash(myValue);

Guarantees

Content-addressing: same content always produces the same hash. The storage layer rejects updates and deletes — merge is a no-op.

Use Cases

  • Audit logs and immutable records
  • Document versioning
  • Certificates and attestations
  • Deduplication (same content = same hash)

Best Practices

Use the Right Type

UnorderedMap
Key-value data, users, items, configs
Vector
Ordered lists, logs, history, queues
Counter
Metrics, totals, distributed counts
LwwRegister
Single values, status, config flags
UserStorage
Per-user data with ownership proof
FrozenStorage
Immutable records, audit logs, certs

Avoid Anti-Patterns

// ❌ BAD — plain object loses concurrent updates
class BadApp { count: number = 0; }

// ✅ GOOD — CRDT handles concurrent updates
class GoodApp { count = createCounter(); }

// ❌ BAD — replacing the CRDT instance loses the ID
const set = profiles.get('alice');
profiles.set('alice', new UnorderedSet()); // ❌

// ✅ GOOD — reuse the existing handle
const s = profiles.get('alice') ?? new UnorderedSet();
s.add('blue'); // ✅
profiles.set('alice', s);
